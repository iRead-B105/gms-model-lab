# GMS Model Lab 작업 지침

이 문서는 저장소 전체에 적용됩니다. 하위 디렉터리에 별도의 `AGENTS.md`가 생기면 해당 디렉터리에서는 더 가까운 문서의 규칙을 우선합니다.

## 프로젝트 목적

GMS 프록시를 통해 이미지·텍스트·TTS 모델의 응답 시간, 토큰 사용량, 크레딧을 비교하는 로컬 테스트 웹입니다.

- 프레임워크: Next.js App Router, React, TypeScript
- UI: Tailwind CSS와 로컬 shadcn/ui 스타일 컴포넌트
- 패키지 관리자: pnpm
- 최소 Node.js 버전: 22.13
- 실행 기록: `data/runs/`
- 생성 이미지: `data/images/`
- 생성 음성: `data/audio/`

실제 GMS 요청은 크레딧을 사용할 수 있습니다. 자동 테스트나 단순 검증을 위해 실제 생성 API를 호출하지 않습니다.

## 주요 폴더

- `app/`: 페이지와 로컬 API Route
- `components/ui/`: 재사용 가능한 기본 UI 컴포넌트
- `features/model-lab/`: 모델 테스트 화면 전용 컴포넌트와 클라이언트 로직
- `lib/types.ts`: 공급자, 모델, 요청, 실행 기록의 공용 타입
- `lib/model-catalog.ts`: GMS 기본 모델 목록, 설명과 안내 크레딧
- `lib/image-sizing.ts`: 이미지 비율과 공급자별 크기 프리셋
- `lib/model-defaults.ts`: 공급자별 기본 파라미터와 직접 입력 시작값
- `lib/server/validation.ts`: 모든 API 입력 검증
- `lib/server/gms/`: GMS 공급자별 호출과 응답 처리
- `lib/server/storage/`: 로컬 실행 기록과 이미지·음성 저장
- `tests/`: 보안, 공급자, 파라미터, 저장소 동작 테스트

공급자 API 로직을 화면이나 Route에 직접 추가하지 않습니다. 공용 규칙은 `lib/`, 서버 전용 로직은 `lib/server/`, 화면 전용 로직은 `features/model-lab/`에 둡니다.

## 개발 명령

```powershell
pnpm install
pnpm dev
pnpm typecheck
pnpm lint
pnpm test
pnpm build
pnpm check
```

코드 변경을 마치기 전 `pnpm check`를 실행합니다. 문서만 변경한 경우에는 문서의 경로·명령·설명이 실제 저장소와 일치하는지 직접 확인합니다.

## 구현 원칙

### 코드와 구조

- TypeScript strict 설정을 유지하고 불필요한 `any`를 추가하지 않습니다.
- import는 가능한 경우 `@/` 별칭을 사용합니다.
- 페이지가 비대해지면 화면 조각은 `features/model-lab/components/`, 순수 계산은 별도 유틸리티로 분리합니다.
- 공급자별 요청 형식은 하나의 거대한 조건문보다 공급자 모듈과 공용 도우미로 분리합니다.
- 패키지를 추가하기 전에 현재 의존성이나 Web API로 해결할 수 있는지 확인합니다.
- 의존성을 변경하면 `package.json`과 `pnpm-lock.yaml`을 함께 반영합니다.
- `.next/`, `next-env.d.ts`, `tsconfig.tsbuildinfo` 같은 생성 파일을 직접 수정하거나 커밋하지 않습니다.

### API와 모델

- 브라우저에서 GMS로 직접 요청하지 않고 항상 로컬 API Route를 거칩니다.
- OpenAI, Gemini, Anthropic의 인증 헤더와 요청 형식을 혼용하지 않습니다.
- 모델 목록은 GMS의 동적 조회 결과를 우선하고, 기본 목록은 조회 실패 시 사용할 최소 안전 목록으로 유지합니다.
- 동적 모델의 단가를 추측하지 않습니다. 근거가 없으면 `미등록`으로 표시하고 실제 잔액 차이를 사용합니다.
- 이미지 비율과 크기는 `lib/image-sizing.ts`에서 관리합니다. 화면과 서버에 크기 목록을 중복 선언하지 않습니다.
- 공급자 기본 파라미터는 `lib/model-defaults.ts`에서 관리합니다. 공급자가 기본값을 선택하도록 둘 수 있는 옵션은 임의 숫자로 고정하지 않고 요청에서 생략합니다.
- 새로운 파라미터는 화면뿐 아니라 서버 검증, 공급자 요청, 로그, 테스트까지 함께 반영합니다.
- 공급자 응답 형식은 항상 검증하고 빈 출력, 부분 출력, 비정상 스트림을 오류로 처리합니다.

### 보안과 개인정보

- GMS 키를 소스, 환경 예제, 테스트 픽스처, 로그, 오류 메시지에 하드코딩하지 않습니다.
- 키를 오류에 포함할 가능성이 있는 값은 응답 전에 마스킹합니다.
- 서버 실행 기록에는 인증 헤더, 키, 토큰, 비밀번호 계열 필드를 저장하지 않습니다.
- 브라우저 키 저장은 사용자가 명시적으로 선택한 경우에만 `features/model-lab/key-storage.ts`를 통해 수행합니다.
- 로컬 저장 키는 암호화되지 않으므로 저장 UI의 공용 PC 경고를 제거하지 않습니다.
- 파일 경로는 허용된 실행 ID와 이미지 파일명만 사용하고 사용자 입력을 그대로 경로에 결합하지 않습니다.
- `data/runs/*.json`, `data/images/*`, `data/audio/*`, `.env*`, 개발 로그는 커밋하지 않습니다.

### 안정성과 비용

- 모든 외부 요청에 취소 신호와 제한 시간을 적용합니다.
- 요청 본문, 스트리밍 프레임, 텍스트 출력, 이미지 크기의 상한을 유지합니다.
- 로컬 결과는 임시 파일 작성 후 교체하는 원자적 저장 방식을 사용합니다.
- 손상된 개별 로그 때문에 전체 기록 조회가 실패하지 않도록 합니다.
- 실제 GMS 호출이 필요한 검증은 예상 크레딧과 호출 범위를 먼저 알리고 사용자의 명확한 요청이 있을 때만 수행합니다.
- 요청 취소는 이미 GMS에 전달된 작업의 과금을 보장하지 않으므로 관련 안내를 유지합니다.

### UI와 문서

- 사용자 화면과 README는 자연스러운 한국어로 작성합니다.
- 코드 식별자와 공급자 API 필드는 원래 영문 명칭을 유지합니다.
- 오류 메시지는 사용자가 다음 행동을 판단할 수 있도록 원인과 해결 방향을 짧게 설명합니다.
- 키, 모델, 프롬프트, 실행 결과처럼 중요한 상태 변경에는 접근 가능한 레이블과 상태 안내를 제공합니다.
- 기능이나 실행 방법이 바뀌면 README와 관련 테스트를 함께 갱신합니다.

## 테스트 기준

변경 범위에 따라 다음 항목을 확인합니다.

- 화면 변경: 타입 검사, 린트, 관련 문구·상태 테스트, 프로덕션 빌드
- API 변경: 정상 구조와 잘못된 JSON·범위·공급자 입력 테스트
- 스트리밍 변경: 정상 완료, 빈 응답, 오류 이벤트, 연결 취소, 크기 제한
- 저장소 변경: 키 마스킹, 안전한 경로, 원자적 저장, 손상 로그 무시
- 이미지 옵션 변경: 비율과 크기 조합, 공급자별 허용값, API에 보내지 않는 내부 필드
- 모델 변경: 공급자별 중복 ID, 동적 조회 실패, 단가 미등록 처리

자동 테스트에서는 실제 GMS 키나 유료 생성 API를 사용하지 않습니다. 로컬 스모크 테스트는 유효하지 않은 입력이 외부 요청 전에 `400`으로 거절되는 범위에서 수행합니다.

## Git 컨벤션

### 브랜치

`main`에서 직접 기능 작업을 누적하지 않고 목적별 브랜치를 사용합니다.

- `feat/<topic>`: 기능 추가
- `fix/<topic>`: 버그 수정
- `refactor/<topic>`: 동작을 유지한 구조 개선
- `docs/<topic>`: 문서 변경
- `test/<topic>`: 테스트 추가·수정
- `chore/<topic>`: 설정, 의존성, 유지보수

브랜치 이름은 영문 소문자와 숫자, 하이픈을 사용합니다. 예: `feat/image-size-presets`, `fix/text-stream-timeout`.

### 커밋

Conventional Commits 형식을 사용합니다.

```text
<type>(<scope>): <subject>
```

허용하는 주요 type:

- `feat`: 사용자 기능 추가
- `fix`: 오류 수정
- `refactor`: 동작 변경 없는 구조 개선
- `docs`: 문서만 변경
- `test`: 테스트 변경
- `style`: 포맷 등 비기능 변경
- `perf`: 성능 개선
- `build`: 빌드·패키지 변경
- `ci`: CI 설정 변경
- `chore`: 그 밖의 유지보수
- `revert`: 이전 커밋 되돌리기

권장 scope는 `ui`, `api`, `gms`, `models`, `storage`, `logs`, `security`, `docs`, `test`, `deps`입니다. 필요한 경우 변경 범위를 더 정확히 나타내는 새 scope를 사용할 수 있습니다.

예시:

```text
feat(ui): 이미지 비율별 크기 프리셋 추가
fix(gms): 텍스트 스트림의 빈 응답 처리
refactor(storage): 실행 로그 저장 로직 분리
docs(readme): 로컬 키 저장 주의사항 보강
test(api): 잘못된 이미지 크기 조합 검증 추가
```

커밋 규칙:

- 제목은 한 가지 변경만 표현하고 마침표를 붙이지 않습니다.
- 기능과 무관한 포맷 변경을 같은 커밋에 섞지 않습니다.
- 변경 이유가 제목만으로 명확하지 않으면 본문에 배경, 선택 이유, 영향을 작성합니다.
- 호환성이 깨지는 변경은 본문 또는 footer에 `BREAKING CHANGE:`를 명시합니다.
- 이슈가 있으면 footer에 `Refs: #123`, 해결하면 `Closes: #123` 형식을 사용합니다.
- 비밀 값, 실제 실행 로그, 생성 이미지, 개인 환경 설정을 커밋하지 않습니다.

### Pull Request

- PR 제목은 커밋 제목과 같은 Conventional Commits 형식을 사용합니다.
- 본문에는 변경 목적, 핵심 변경, 검증 결과를 작성합니다.
- UI 변경은 필요할 때 변경 전후 이미지나 짧은 설명을 포함합니다.
- API·보안·비용 관련 변경은 호환성, 키 저장 여부, 실제 호출 여부를 명시합니다.
- `pnpm check`가 통과하지 않은 상태로 병합하지 않습니다.
- 큰 리팩터링과 사용자 기능 변경은 가능한 한 별도 PR로 분리합니다.

## 완료 조건

- 요청한 동작이 구현되고 관련 예외 상황이 처리되어 있습니다.
- 키와 로컬 데이터가 의도하지 않은 위치에 저장되지 않습니다.
- 코드 구조와 문서가 실제 동작과 일치합니다.
- 관련 테스트가 추가되거나 갱신되어 있습니다.
- `pnpm check`가 통과합니다.
- 커밋과 PR이 위 Git 컨벤션을 따릅니다.
